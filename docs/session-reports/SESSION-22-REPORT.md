## Hurdles + solutions

### 1. M5 crash-catch (the Daisy save)

Mid-plan-review for #116, I missed a cascading failure: if the daemon stubs candidates with NULL `proposed_name`/`proposed_steps`, then `apply_curriculum_task` (which atomically fires `promote_candidate_to_skill` inside the same SQL tx) would abort on the NOT-NULL constraint at line 295 of `scripts/012_sleep_learning.sql`. Daisy caught it. **Fix:** documented the compose-before-apply mandate in both `compose_skill_candidate` and `apply_curriculum_task` tool descriptions and in ARCHITECTURE.md §M5. The live test (D3) validated the path works precisely because compose was called first.

### 2. Security hook false positive on the regex match method

The `security_reminder_hook.py` PreToolUse hook matched a literal substring in `scripts/lint-boundaries.ts` and blocked the `Write` call, mistaking the `RegExp.prototype` match method for `child_process` shell invocation. **Fix:** refactored the scanner loop to `String.prototype.matchAll()` — cleaner code anyway, no shell-injection surface, hook satisfied.

### 3. Windows UNC long-path prefix in fence output

`glob` on Windows returned `//?/C:/...` paths that `path.relative()` couldn't normalize. **Fix:** explicit prefix strip (`v.file.replace(/^[\\/]{2}\?[\\/]/, "")`) before computing the relative path. Output is now clean cross-platform.

### 4. MCP server cold-start = stale tool registry

After shipping `compose_skill_candidate` in `dist/index.js`, `ToolSearch` couldn't find it — the running MCP server child process was spawned at session start with the pre-S22 build. **Workaround during the live test:** direct `supabase.from('skill_candidates').update(...)` — functionally identical (same columns, same `state='mined'` guard, same `model='orchestrator:claude'` stamp). **Permanent resolution:** session restart surfaced the tool; smoke-tested against fresh candidate #9 with identical result. Lesson captured in LOG 11540: *shipping a new MCP tool is a two-phase deploy — code merge + session restart*.
