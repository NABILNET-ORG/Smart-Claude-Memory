## 1. Problem Statement

The M8.2 D3.js force-graph GUI is **hard-clamped at 200 nodes / 500 edges** ([src/gui/server.ts:97-102](../../src/gui/server.ts#L97-L102)) because client-side force simulation collapses in browser frame-time past that envelope. Defaults are even more conservative (60/120).

`kg_nodes` is **already populated faster than the GUI can show**. With one large repo indexed (8086 memory_chunks today), kg_nodes will cross 10k within a year of usage and 50k+ at multi-project scale. The current cap means **the operator can only ever see a tiny biased sample** — whichever `k` rows the `list_kg_nodes` tool returns first. The graph view becomes a toy, not a navigation surface.

**M8.3 goal:** server-side semantic clustering so the browser receives at most ~200 "Super Nodes" at any zoom level, regardless of how many real nodes exist below. Drilling into a Super Node fans out to its children (also clustered if still >200). The renderer never chokes.

---

## 2. Non-Goals (Scope Fence)

- **No replacement for `list_kg_nodes`.** That tool stays exact/raw. Clustering is a NEW surface (`/api/graph/clusters`), not a mutation of the existing one.
- **No new pgvector extension.** Supabase managed disallows `plpython3u`, so pure-SQL K-Means via PL/Python is out. Compute happens in Node, results land in a new table.
- **No real-time re-clustering.** Cluster assignments are batch-computed by a daemon on a cadence (default: hourly when dirty). The GUI reads stable cluster_ids.
- **No multi-project mixing.** Clustering is per-`project_id`. Cross-project visualization is M8.4 territory.
- **No UMAP/t-SNE coordinate projection in v1.** Layout stays D3-force-driven at the Super Node level. 2D embedding projection deferred to v0.2.

---

## 3. Sovereign Constraints (HARD)

1. **750-line code ceiling** — split clustering daemon, query layer, and HTTP route into 3 modules.
2. **No new Postgres extensions.** All k-means / HNSW post-processing happens in Node against pgvector output.
3. **HNSW index already exists** on `kg_nodes.embedding` ([scripts/020_knowledge_graph.sql:53-54](../../scripts/020_knowledge_graph.sql#L53-L54)) with `vector_cosine_ops`. Reuse it — do NOT add a parallel index.
4. **`pgvector` lives in the `extensions` schema.** Every new RPC touching vector types MUST `set search_path = public, extensions, pg_catalog` — same fix as SESSION-17 (ERROR memory 11507).
5. **Additive schema only.** New tables, new RPCs, new routes. Zero migrations to kg_nodes/kg_edges. The existing M8.2 `/api/graph` endpoint stays byte-compatible.
6. **Daemon throttled via ARM.** The clustering daemon runs LLM-free (no Ollama embed calls — embeddings already exist on kg_nodes) but still uses `checkDaemonBudget("clustering_scanner", "ollama_calls", 0)` to register its presence in `system_dashboard`, matching the v2.2.2 ARM contract.

## 4. Approach Analysis — K-Means vs. HNSW Communities vs. Hybrid

### 4.1 Option A: K-Means in Node (mini-batch)

- **How:** pull all `kg_nodes.embedding` rows for project, run k-means in JS (e.g., `density-clustering` or hand-rolled mini-batch), write cluster_id back via bulk UPSERT.
- **Pros:** simple, deterministic with seeded RNG, well-understood, K is a knob.
- **Cons:** K must be chosen up front (poor at non-spherical manifolds). Embedding-space cosine distance is fine but k-means is L2-native; need normalization. Cluster IDs are unstable across re-runs (label-shift). Bulk-pull of 50k × 768d = ~150 MB into Node memory per project — manageable but not free.
- **Cost:** O(N · K · iterations · d). At N=50k, K=√N≈224, 10 iters, d=768 → ~85B ops, ~30 s single-thread.

### 4.2 Option B: HNSW kNN graph + Louvain communities

- **How:** for each kg_node, fetch its top-15 cosine neighbors via existing HNSW index (one RPC, projected down to (source_id, target_id, similarity)). Build an in-memory kNN graph in Node, run Louvain community detection (e.g., `graphology-communities-louvain`). Each community becomes a cluster.
- **Pros:** respects manifold structure (K-Means doesn't). Reuses existing HNSW index — no fresh compute on embeddings. Cluster IDs naturally stable when graph is mostly unchanged (incremental Louvain is well-studied). Cluster sizes adapt to data density.
- **Cons:** N HNSW lookups = N round trips OR a single `select … order by embedding <=> e limit 15` per node. Either way, network-bounded. Louvain is non-deterministic without a fixed seed.
- **Cost:** one batch RPC returning all (src, tgt, sim) triples (~750k rows at N=50k, k=15) is ~50 MB raw. Louvain on 50k nodes, 750k edges runs in ~5 s in JS.

### 4.3 Option C: Hybrid — Hierarchical (RECOMMENDED)

Combine both at different levels:

- **Coarse level (Super Nodes shown at zoom-0):** K-Means with K=√N (sqrt-N rule, classic). Fast, deterministic, good enough for a "where are the dense regions" overview.
- **Fine level (drill-down into one Super Node):** Louvain on the HNSW kNN sub-graph restricted to nodes inside that Super Node. Reveals the local community structure that K-Means averages over.
- **Why hybrid wins:** the operator's UX is two-step — "show me the shape" → "show me what's in this region." Different algorithms are best at each. We compute both at daemon time and store both columns; the route picks one based on `?zoom=`.

Each kg_node ends up with `(supernode_id, community_id)`. Two indexes, two cheap groupings, one D3 view per zoom.

## 5. Schema — `scripts/023_kg_clustering.sql`

```sql
-- M8.3 semantic clustering side-table. Additive — kg_nodes is untouched.
create table if not exists public.kg_node_clusters (
  project_id      text   not null,
  node_id         bigint not null references public.kg_nodes(id) on delete cascade,
  supernode_id    int    not null,   -- coarse, K-Means, K = ceil(sqrt(N))
  community_id    int    not null,   -- fine, Louvain on HNSW kNN sub-graph
  computed_at     timestamptz not null default now(),
  primary key (project_id, node_id)
);

create index if not exists kg_node_clusters_super_idx
  on public.kg_node_clusters (project_id, supernode_id);
create index if not exists kg_node_clusters_community_idx
  on public.kg_node_clusters (project_id, supernode_id, community_id);

alter table public.kg_node_clusters enable row level security;
-- Service-role only (matches project_supabase.md pattern).
revoke all on public.kg_node_clusters from anon, authenticated;

-- Summary view: one row per (project, supernode) with mean position placeholder,
-- node count, and the 3 most-frequent labels (for the Super Node display label).
create or replace view public.kg_supernodes as
select
  c.project_id,
  c.supernode_id,
  count(*)::int                                       as node_count,
  (
    select array_agg(label order by freq desc)
    from (
      select n.label, count(*) as freq
      from public.kg_nodes n
      join public.kg_node_clusters c2 using (id)  -- alias via implicit; refined in actual impl
      where c2.project_id = c.project_id and c2.supernode_id = c.supernode_id
      group by n.label
      order by count(*) desc
      limit 3
    ) t
  )                                                   as top_labels,
  max(c.computed_at)                                  as computed_at
from public.kg_node_clusters c
group by c.project_id, c.supernode_id;
```

> **Note:** the view body above is a sketch; the actual JOIN against kg_nodes needs explicit aliasing. Lock during Task 1.

### kNN edges (in-memory only — NOT persisted)

The HNSW kNN graph used to drive Louvain is consumed once per daemon run and discarded. We do NOT persist it; that would duplicate kg_edges. The single RPC below returns it on demand.

### New RPC — `kg_knn_pairs`

```sql
create or replace function public.kg_knn_pairs(
  p_project_id text,
  p_k          int default 15,
  p_min_sim    double precision default 0.5
) returns table (source_id bigint, target_id bigint, similarity double precision)
language sql
stable
security definer
set search_path = public, extensions, pg_catalog
as $$
  with src as (
    select id, embedding from public.kg_nodes
    where project_id = p_project_id and embedding is not null
  )
  select s.id as source_id, n.id as target_id,
         1 - (s.embedding <=> n.embedding) as similarity
  from src s
  cross join lateral (
    select id, embedding from public.kg_nodes
    where project_id = p_project_id and embedding is not null and id <> s.id
    order by s.embedding <=> embedding
    limit p_k
  ) n
  where 1 - (s.embedding <=> n.embedding) >= p_min_sim;
$$;
```

Each lateral subquery hits the HNSW index. At N=50k, k=15, this returns ≤750k rows in a single round trip. Bench target: <10 s end-to-end at 50k.

## 8. HTTP API

### `GET /api/graph/clusters?level=super`

Returns Super Node graph (one node per supernode). Edges aggregate kg_edges by `(src.supernode_id, tgt.supernode_id)` with `count(*)` as weight.

```json
{
  "level": "super",
  "project_id": "claude-memory",
  "nodes": [
    { "id": "S:0", "supernode_id": 0, "label": "M8.1 KG", "node_count": 412 }
  ],
  "edges": [
    { "source": "S:0", "target": "S:1", "weight": 38 }
  ],
  "computed_at": "..."
}
```

### `GET /api/graph/clusters?level=drill&supernode_id=N`

Returns the actual kg_nodes inside one supernode, plus their kg_edges restricted to in-cluster. If member count > `GRAPH_NODE_LIMIT_MAX` (200), responds with a recursive cluster view by community_id ("nested Super Nodes").

### Pagination & clamps

Same clamp constants as the existing `/api/graph` route. Super Node level rarely exceeds 200 (K=√N caps it at √50k ≈ 224, with a hard ceil at 200).

## 9. Daemon Topology

Same shape as the v2.2.2 ARM-aware daemons (`graph_extractor`, `graduation_scanner`, etc.):

- Interval: `SCM_CLUSTERING_INTERVAL_MS`, default `1_800_000` (30 min).
- Batch: one project per tick (round-robin across known project_ids).
- Dirty check: `(count(kg_nodes) - count(kg_node_clusters))` > 0 OR `max(kg_nodes.updated_at) > max(kg_node_clusters.computed_at)`.
- ARM gate: `checkDaemonBudget("clustering_scanner", "ollama_calls", 0)` — registers presence, never blocked (no LLM calls).
- Telemetry: emits `daemon_run` event with `{kmeans_k, supernodes_created, louvain_communities, embeddings_loaded, duration_ms}` for `system_dashboard`.
- Retention: `kg_node_clusters` rows live as long as the kg_node they reference (`on delete cascade`). No separate sweep.

## 12. Open Questions

1. **Louvain seeding** — JS implementations differ in determinism. May need to ship our own or fork `graphology-communities-louvain` to expose a seed param. Decision pending Task 2 spike.
2. **Cluster ID stability across re-runs** — K-Means label-shift is the classic problem. Option: align new run's centroids to old via Hungarian matching. Cost is O(K²); at K=200 trivial. Locks UI stability when zoom level doesn't change.
3. **Should Super Node edges aggregate ALL kg_edges or only "between supernodes"?** Spec currently says aggregate cross-supernode only. Same-supernode edges become hidden until drill-down. Confirm during Task 4.
4. **GUI: 2D position for Super Nodes** — v1 says "let D3 force-graph place them." Future v0.2 could pre-compute UMAP coords server-side for stability across reloads. Defer.
5. **Centroid embeddings** — should we persist a centroid vector per supernode? Enables "find Super Node nearest to query embedding" semantic search. Cheap (one row per K=200). PUNT to v0.2.

## 14. Self-Review

**Why hybrid over pure K-Means or pure Louvain?**
K-Means is fast but ignores manifold structure; Louvain respects it but is overkill for a 50k-row overview. Two-level lets us pay each cost only where its property matters. The operator never sees both layers at once — the GUI swaps in based on zoom.

**Why no UMAP/t-SNE?**
Adds a Python dep (umap-learn) or a wasm port; both balloon scope. D3 force-graph at the Super Node level is already the existing render path. No new tech needed for v1.

**Risk: kNN RPC scales as N²/k in worst case.**
The HNSW index keeps the per-row lookup at O(log N), so total is O(N log N). At N=50k that's manageable. Above N=100k, we shard by project and process in chunks.

**Risk: cluster_id instability bothers users.**
Mitigation in section 12 Q2. Even without alignment, the GUI labels Super Nodes by `top_labels`, not by ID, so UI text stays sensible across reruns.

**Locked decisions (Session 40):**
- Two-level hybrid (K-Means coarse, Louvain fine).
- New side-table `kg_node_clusters` — additive only.
- New RPC `kg_knn_pairs` — reuses existing HNSW index.
- Existing `/api/graph` endpoint stays byte-compatible.
- Daemon throttled via ARM (registers, never blocks; no LLM calls).
